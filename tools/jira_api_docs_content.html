To use OAuth 2.0 authentication (recommended by InfoSec)

https://developer.atlassian.com/cloud/jira/platform/rest/v3/intro#authentication

https://developer.atlassian.com/cloud/jira/platform/oauth-2-3lo-apps/

Please follow the steps as described in the "oauth-2-3lo-apps" doc to set up your OAuth authentication. But please note the following details which are not fully explained in the doc:

  1. When you are in Developer Console and in the "Permission" page to pick the scopes, please only choose the section "Jira API" by clicking on its "Add" button and then proceed to click its "Configure" button to choose its scopes. If you accidentally add other products (such as "User Identity API"), it would confuse the "Authorization URL generator" and then it would give you the wrong generated URL. Please refrain from doing that. If you accidentally add the wrong scopes form unrelated products, please remove them from "Permission" page using the "Configure" button.

    1. There is a known issue/bug in developer console: If you select items from multiple "API name" items in the list in "Permission" tab, any further calls to REST API would always return empty result. Therefore, for calling Jira REST API, you should only select permissions from the API name "Jira API" in permission setting stage of development console.

  2. Note that if the user is doing it with Jira Cloud site as a testbed, the user needs to provide the complete Callback URL such as https://trendmicro.atlassian.net/jira/your-work but not just the base URL such as https://trendmicro.atlassian.net/

    1. The original design of OAuth 2.0 is that the user's site will send the request to https://auth.atlassian.com and it would callback to user's own site and return with a "?code=xxx" so the user's site can capture it.

    2. This is because when logging in to Jira Cloud (to the base URL), Jira will automatically redirect to the user's default page.

    3. In the following OAuth 2.0 set up process, it will append a "?code=xxxx" query string parameter to the result callback URL.

    4. But if you only provide the simple base URL, Jira will redirect again to the user's default page and thus the "?code=xxxx" info is lost.

  3. Also note that in the doc it uses the "curl" tool. In the examples the single quote (') is used for quoting the texts. However, single quotes cannot be used in Windows cmd.exe window.

    1. Either do the steps in a Linux / Mac environment. Or

    2. In the Windows environment, do it in the Cygwin window.

  4. After you obtained the "authorization code" and "access token", you need to proceed to the following steps (authorization code => access token => cloud-id) in an hour. If you just keep the info but procrastinate for too long, it will expire and then you need to start over again.

  5. Remember at the last step to use the cloud-id, the "Authorization" header and the access token (which token type is "Bearer") still needs to be provided in REST API code.

  6. Please note that the authorization code is valid for only 1 hour, and the access token will be expired in 1 hour as well. In order to use access token for longer than 1 hour:

    1. In the first step to retrieve the authorization code, add an "offline_access" value to the "scope" parameter when using the authorization URL.

    2. Each time when the current access token is older than 1 hour, your code needs to send the refresh token request to auth.atlassian.com so it can return with a new pair of access token and refresh token, and the used refresh token is disabled after its usage.  See also: https://developer.atlassian.com/cloud/jira/platform/oauth-2-3lo-apps/#faq1    => How do I get a new access token, if my access token expires or is revoked.

Using OAuth 2.0 authentication is preferred because the generated tokens are automatically expired with a given period. (default: unused expire in 3 months and all expire in 1 year)

Note that for all the following "methods" if OAuth 2.0 is used, use the "https://api.atlassian.com/ex/jira/{cloud-id}" as the actual method URL base, for example "https://api.atlassian.com/ex/jira/{cloud-id}/rest/api/3/issue/{issueIdOrKey}" . The cloud-id is what you obtain from the above "oauth-2-3lo-apps" doc in the final step. 

To authorize with API token

Go to Jira cloud site, open avatar => Manage account => Security => API token / Create and manage API tokens => Create API Token

Type a label, for example "For testing REST API". However, it is strongly recommended to give it a more specific label which explains the purpose of such token.

Choose an expiry date. The default would be 7 days from the current date, and the maximum is 1 year from that.

Click “Create” => Copy the newly generated token value.

Paste the text string to a text file and save it to a safe place for later use. (this is important because it can only copied once)

Click "Close"

Use the generated text string in the HTTP request Authorization header in the format: "Basic xxxxxxxxx"

  • note that in the Jira Cloud, you should use the email address as account name before generating the basic token like this: "Basic " + BASE64($email + ":" + $api_token)

This method is less favorable because the token will never be expired and InfoSec has concern about that.

Note that if this authorization approach is used, the base URL of the REST API would remain as the Jira Cloud site base URL, such as "https://trendmicro.atlassian.net/rest/api/3/issue/{issueIdOrKey}"

Basic authentication no longer allowed

Originally you could use the "basic authentication" method by combining the account name and password this way

  • Authorization: "Basic " + BASE64($account + ":" + $password)

However, such methodology was deprecated and removed since 2019, now its usage is totally blocked. See also the original announcement:

https://developer.atlassian.com/cloud/confluence/deprecation-notice-basic-auth/

Search for issues

https://developer.atlassian.com/cloud/jira/platform/rest/v3/api-group-issue-search/#api-rest-api-3-search-get
Method: GET /rest/api/3/search

example:
https://trendmicro.atlassian.net/rest/api/3/search?jql=project=PROJ

Remember to use pagination query string parameters such as "startAt" / "maxResult" as well as the "fields" parameter to limit the size of outputs or else it might exceed limitation.

Get issue

https://developer.atlassian.com/cloud/jira/platform/rest/v3/api-group-issues/#api-rest-api-3-issue-issueidorkey-get
Method: GET /rest/api/3/issue/{issueIdOrKey}

example:
https://trendmicro.atlassian.net/rest/api/3/issue/PROJ-123

Remember to use the "fields" parameter to select only the fields need to be returned.

Create issue

Get "createmeta" info first

https://developer.atlassian.com/cloud/jira/platform/rest/v3/api-group-issues/#api-rest-api-3-issue-createmeta-get
Method: GET /rest/api/3/issue/createmeta

First call the "create meta" to get the allowed issue types:

Method: GET /rest/api/3/issue/createmeta?projectKeys=(project-key)

example:
https://trendmicro.atlassian.net/rest/api/3/issue/createmeta?projectKeys=PROJ

Parse the createmeta info for available issue types. Then use the needed issue type to get the available fields / required fields:

Method: GET /rest/api/3/issue/createmeta?projectKeys=(project-key)&issuetypeNames=(issue-type-name)&expand=projects.issuetypes.fields

example:
https://trendmicro.atlassian.net/rest/api/3/issue/createmeta?projectKeys=PROJ&issuetypeNames=Escalation&expand=projects.issuetypes.fields

Parse the second createmeta info for available fields in the create state of a new issue and also find out what are the required ones.

Create the issue after createmeta info is parsed

https://developer.atlassian.com/cloud/jira/platform/rest/v3/api-group-issues/#api-rest-api-3-issue-post
Method: POST /rest/api/3/issue

(see the document in above link for details on filling the POST request payload body)

Update / edit issue

Get "editmeta" info first

Appendix for OAuth 2.0 authorization process

The authorization process is slightly more complicated than the normal (basic) authorization. The following pseudo code / steps aim to help explaining how it works.

First of all, please bear in mind that it is to grant the authorization between the third-party application/site (APP) and Jira Cloud site (JIRA). The following explanation uses USER / APP / JIRA as legends thusly.

  • USER: https://developer.atlassian.com/

to enable / set up the OAuth 2.0

  • USER: https://auth.atlassian.com/authorize?audience=api.atlassian.com&client_id={CLIENT-ID}&scope={SCOPES}&redirect_uri={CALLBACK-URL}&state=${YOUR_USER_BOUND_VALUE}&response_type=code&prompt=consent

The above URL can be copied from the Developer console / Authorization page. Note that the CALLBACK-URL should be the URL of the APP.

  • JIRA => APP: GET https://APP/jira/your-work?code={AUTH-CODE}&state=%24%7BYOUR_USER_BOUND_VALUE%7D

Note that the "YOUR_USER_BOUND_VALUE" actually should be a user ID your APP site can associate to the user, which should be entered in JIRA's Developer console and be recognized by your APP site, normally an email address. But if your APP sites uses certain ID or even just ignore it, it still works.

The APP site should extract the AUTH-CODE from "code=" parameter for such user, and then pops up a window asking the user to provide the client ID and client secret, which are available in Developer console / Settings page.

  • APP => JIRA: POST https://auth.atlassian.com/oauth/token 

  • Payload: {"grant_type": "authorization_code","client_id": "{CLIENT-ID}","client_secret": "{CLIENT-SECRET}","code": "{AUTH-CODE}","redirect_uri": "{CALLBACK-URL}"}'

  • JIRA => APP: {"access_token":"{ACCESS-TOKEN}","scope":"{SCOPES}","expires_in":3600,"token_type":"Bearer"}

The app should extract the access token for later usage, it will expire in 1 hour (but it should be long enough to execute all the consequent REST API calls in the "batch").

  • Extra step only for the first time, in order to obtain the client-id which is the ID list of the target Jira Cloud sites

  • APP => JIRA: GET https://api.atlassian.com/oauth/token/accessible-resources

  • Header: Authorization: Bearer {ACCESS-TOKEN}

  • Header: Accept: application/json

  • JIRA => APP: [{"id":"{CLOUD-ID1}","url":"{URL1}","name":"{NAME1}","scopes":[{SCOPES}],"avatarUrl":"{AVATAR-URL1}"} ...]

The reason for doing this is because it is possible to register the same OAuth APP for multiple Jira sites, thus we need to know the ID of each Jira site. The result of this REST API call is a list of available IDs, each would map to a Jira Cloud site such that OAuth authorization can access to. The APP should memorize the mapping for following step, which is the final actions your app normally would do to communicate with Jira. This final step can execute many REST API calls for querying / updating Jira data, for as long as the access token is still valid.

  • Do a REST API call within the "batch" (all GET / PUT / POST methods are available depending on the needs)

  • APP => JIRA: GET https://api.atlassian.com/ex/jira/{CLOUD-ID}/rest/api/3/search?jql=project=PROJ

  • Header: Authorization: Bearer {ACCESS-TOKEN}

  • Header: Accept: application/json

  • When done, loop to do another request with the access token.

Note that all the above should be done within an hour. If your code needs to continuously call REST API with the access token for more than 1 hour, please obtain a "refresh token" for such purpose: https://developer.atlassian.com/cloud/jira/platform/oauth-2-3lo-apps/#faq1

How to delete OAuth 2.0 app when no longer needed

Log in to Jira cloud site

Avatar => Manage account => Connected apps / Apps with access to your accounts

Choose the "app" and then click its "Remove access"

Log in to Developer console: https://developer.atlassian.com/

Avatar => Developer console => select app => Settings => Delete app => Delete

How to delete OAuth 2.0 app which were added/installed after Feb 2024

Note that starting from Feb 2024, after “Remove access” step the app can still not be deleted from developer console, which is due to the following change in Jira:

https://developer.atlassian.com/changelog/#CHANGE-1409

Although the change is to fix permission-grant sync process out of sync problem, it brings a new problem that the site no longer has the authorization info and thus the app needs to be explicitly found at the site first and uninstall it from there. These are the steps for apps which were added/installed after Feb 2024 (and hence having the above problem):

  1. USER: Just in case, the user still needs to remove access:

    1. Log in to Jira cloud site

    2. Avatar => Manage account => Connected apps / Apps with access to your accounts

    3. Choose the “app” and then click its “Remove access”

  2. USER: Log in to Developer console: https://developer.atlassian.com/

    1. Avatar => Developer console => select the app => Settings

    2. Click on “Delete app” and then click “Delete” in the pop up window.

    3. If it is successful, it’s done. If it shows an error about the app being used and thus is locked, proceed to the following steps.

    4. Copy the value from “Client ID”

    5. file an ETSSD Jarvis helpdesk ticket, and paste the client ID in the ticket. Request the ticket to be handled by Jira’s organization admin.

  3. Jira System Org Admin: Log in to https://admin.atlassian.com

    1. Compose the URL to access the app on the site: https://admin.atlassian.com/s/{SITE_ID}/user-connected-apps/{CLIENT_ID}

      1. Note that for production Jira site: SITE_ID=79a3ee80-0d14-4a82-9335-03f989902e7a

      2. For sandbox Jira site: SITE_ID=440a7ad1-478f-4a05-9f98-bc29f46b75b5

      3. For other sites, use “Products” tab in admin.atlassian.com to find their ID

      4. This is how to compose the URL:

        1. Copy the aforementioned URL template to any text editor: https://admin.atlassian.com/s/{SITE_ID}/user-connected-apps/{CLIENT_ID}

        2. Copy the site ID mentioned above

        3. Paste and replace the {SITE_ID} in the editor

        4. Copy the client ID as reported by the user

        5. Paste and replace the {CLIENT_ID} in the editor

        6. The editor now contains the newly composed URL, copy it and proceed the following steps

    2. Paste the composed URL to the browser address box, which should show the app info in that site, where it should have an “Uninstall” button on the top-right corner, click to uninstall it.

    3. Notify the user it’s now uninstalled.

  4. USER: Log in to Developer console: https://developer.atlassian.com/

    1. Avatar => Developer console => select the app => Settings

    2. Delete app => Delete

The service account readiness requirement

Service account needs to be enrolled with MS Authenticator first for SSO login.
It needs to be logged in at least once, and then its account status will change from "invalid" to "valid".
Once valid, it can be used for calling Jira Cloud's REST API.

Beware of rate limiting

Jira Cloud limits how many requests can be sent to a site per minute, This is a setting controlled by Atlassian and we have no technical details about the exact figuration. When such limit is reached, your REST API call would be rejected with a return code 429 and a return message something like:

{"code":429,"message":"Request has been rate limited"}

Your code can only call the REST API again after certain amount of time. If your code immediately attempts to call again, it will still be blocked but will have a longer "penalty waiting time" for the next available/allowed time. In order to get the next available time, your code should try to parse the header  X-RateLimit-Reset (which provides the next time the call will be allowed) or Retry-After (the number of seconds your code should wait). The details can be found here:

https://developer.atlassian.com/cloud/jira/platform/rate-limiting/

In order to avoid getting the #429 error code too often (because retrying to call REST API is a waste of network bandwidth), it is wise to insert interval time between your calls for at least 5 seconds or more.
